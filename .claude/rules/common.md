# Common Rules — All Languages

These rules apply to every file generated by every agent, regardless of language.

## Naming
- Files: `kebab-case.ts`, `kebab-case.py`, `kebab-case.go`
- Classes / Types: `PascalCase`
- Functions / Variables: `camelCase` (JS/TS), `snake_case` (Python/Go)
- Constants: `UPPER_SNAKE_CASE`
- Boolean variables: prefix with `is`, `has`, `can`, `should` — e.g. `isLoading`, `hasError`

## File Structure
- One primary export per file (one class, one component, one function group)
- Imports ordered: stdlib → third-party → internal (with blank line between groups)
- Max file length: 300 lines — extract if longer
- Max function length: 50 lines — extract helpers if longer
- Max line length: 100 characters

## Comments
- No commented-out code — delete it, git history preserves it
- No `// TODO` without an issue reference: `// TODO(#123): fix this`
- Comments explain *why*, not *what* — the code explains what
- JSDoc / docstrings only on public APIs and exported functions

## Error Handling
- Never swallow errors silently: no empty `catch {}` blocks
- Always log the original error before re-throwing or transforming
- Distinguish recoverable errors (retry) from unrecoverable (fail fast)
- User-facing errors must be human-readable — never expose stack traces or DB errors

## Security (Universal)
- No hardcoded secrets, API keys, tokens, or passwords — ever
- No `eval()` or dynamic code execution from user input
- Validate and sanitize all external inputs at system boundaries
- Principle of least privilege — request only the permissions needed

## Testing
- Every public function must have at least one test
- Test file co-located or in parallel `tests/` directory
- Describe block names: noun phrases ("UserService")
- It block names: complete sentences ("returns null when user not found")
- No `console.log` in test files — use assertions

## Git
- Commit messages: `type(scope): short description` — e.g. `feat(auth): add OAuth login`
- Types: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `perf`, `ci`
- One logical change per commit
- Never commit directly to `main` — always use feature branches

## Performance
- No synchronous I/O in hot paths (no `fs.readFileSync` in request handlers)
- No N+1 queries — batch database calls
- No unbounded loops over large datasets — paginate or stream
- No blocking the event loop for > 10ms
